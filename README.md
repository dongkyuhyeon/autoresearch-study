# 구조물 안전성 물리 추론 AI 경진대회 실험 기록

## final ranking 44/484

## 1. 대회 개요

이번 프로젝트는 **구조물 안전성 물리 추론 AI 경진대회**를 준비하면서 수행한 실험 과정과 시행착오, 그리고 최종적으로 얻은 성능 개선 결과를 정리한 문서입니다.  
대회의 핵심 과제는 이미지로부터 구조물의 안정성을 판단하는 것이었고, Karpathy/autoresearch Agent를 사용했습니다.

### 데이터 구성

- **train**: 정적(static) 이미지 데이터
- **dev / test**: 동적(dynamic) 이미지 데이터
- 모든 데이터는 **이미지 기반 데이터셋**으로 구성

즉, 학습에는 상대적으로 정적인 환경의 이미지가 주어졌고, 실제 성능을 평가하는 dev/test는 더 다양한 시점, 조명, 장면 변화가 포함된 **동적 도메인**이었습니다.  
따라서 이번 대회에서는 단순히 train 성능을 높이는 것보다, **정적 학습 데이터와 동적 평가 데이터 사이의 도메인 차이(domain gap)를 얼마나 잘 줄이느냐**가 핵심이었습니다.

---

## 2. 전체 실험 흐름 요약

실험은 크게 다음 흐름으로 진행되었습니다.

1. **dev 분할 비율 탐색**  
   동적 dev 데이터를 일부 학습에 사용하고, 나머지를 hold-out 검증용으로 두었을 때 어떤 비율이 가장 잘 일반화되는지 확인

2. **seed 고정 실험**  
   작은 dev 데이터셋 분할에서 발생하는 우연성과 분산을 줄이기 위해 seed별 결과를 비교하고, 이후 실험의 기준 seed를 결정

3. **synthetic 데이터 증강 실험**  
   train 이미지에 dev/test 조건을 반영한 synthetic 이미지를 점진적으로 추가하여 도메인 적응 성능 향상 여부를 검증

이 흐름은 단순히 성능 숫자만 비교한 것이 아니라,

- 어떤 split이 더 믿을 만한지,
- 어떤 seed가 더 안정적인지,
- synthetic 데이터가 실제로 dev/test 일반화에 도움이 되는지

를 순서대로 검증하는 과정이었습니다.

## 3. 실험 1: dev 분할 비율 비교

### 목적

처음 학습할 때는 동적 이미지를 더 정확하게 맞추기 위해, `joe_baseline_code_upgrade` 기반으로 dev 데이터를 학습용 일부와 hold-out 검증용 나머지로 나누어 사용했습니다.  
핵심 질문은 다음과 같았습니다.

> **dev를 어느 정도 비율로 학습에 포함시켜야 동적 도메인 적응과 검증 신뢰성을 동시에 확보할 수 있는가?**

비교한 설정은 다음 세 가지였습니다.

- 20/80
- 30/70
- 40/60

여기서 앞 숫자는 `train + dev 일부`를 이용한 학습 비중, 뒤 숫자는 `hold-out 검증용 dev` 비중입니다.

---

### 실험 결과

- **Experiment 1 (20/80)**  
  - best val_logloss = **0.2291**
  - 최종 dev_full_logloss = **0.185459**

- **Experiment 2 (30/70)**  
  - best val_logloss = **0.1701**
  - 최종 dev_full_logloss = **0.120270**

- **Experiment 3 (40/60)**  
  - best val_logloss = **0.2304**
  - 최종 dev_full_logloss = **0.160855**

### 결론

세 실험 중 **Experiment 2 (30/70)** 이 가장 좋은 결과를 보였습니다.

---

### 해석

#### 왜 30/70이 가장 좋았는가?

`train 1000 + dev 30` 설정은 동적 도메인 적응에 필요한 최소한 이상의 dev 정보를 학습에 제공하면서도, 동시에 `dev 70`이라는 비교적 충분한 hold-out 검증 세트를 유지할 수 있었습니다.

즉,

- 학습 측면에서는 동적 데이터 특성을 어느 정도 반영할 수 있었고,
- 검증 측면에서는 checkpoint 선택에 필요한 일반화 판단 근거를 충분히 확보할 수 있었습니다.

#### 왜 20/80은 상대적으로 불리했는가?

- dev 20은 학습에 넣기에는 너무 적었습니다.
- 동적 도메인 적응을 하기에는 정보량이 부족했습니다.
- hold-out은 넉넉했지만, 정작 모델이 동적 데이터 특성을 배우는 데 한계가 있었습니다.

#### 왜 40/60은 기대보다 덜 좋았는가?

- dev 40은 학습에는 더 도움이 될 수 있습니다.
- 하지만 hold-out 크기가 줄어들면서 checkpoint 선택의 신뢰성이 떨어질 수 있습니다.
- 작은 dev 환경에서는 검증셋이 지나치게 줄어들면, 오히려 generalization 판단이 불안정해질 수 있습니다.

### 정리

이 실험을 통해, 단순히 dev를 많이 학습에 넣는다고 항상 좋은 것이 아니라 **학습 적응 효과와 검증 신뢰성 사이의 균형**이 중요하다는 점을 확인했습니다.  
이후 실험들은 **30/70 분할 전략**을 기본 방향으로 삼았습니다.

## 4. 실험 2: seed 고정 및 분할 안정성 검증

### 목적

다음 단계 목표는 **동적 데이터를 더 잘 맞추기 위한 데이터 증강**이었습니다.  
하지만 증강 실험을 진행하기 전에 먼저 확인해야 할 것이 있었습니다.

> **현재 성능 차이가 정말 augmentation 때문인지, 아니면 seed에 따라 달라진 분할 운 때문인지 구분할 수 있는가?**

그래서 증강 실험에 들어가기 전, seed를 고정하기 위한 검증을 먼저 수행했습니다.

---

### 왜 seed를 고정해야 했는가?

#### 1. 데이터 분할 결과가 seed에 따라 달라지기 때문

특히 dev 100처럼 작은 데이터셋을 30/70으로 나누면,

- 어떤 샘플이 train 쪽으로 들어가고
- 어떤 샘플이 hold-out으로 들어가는지

가 seed에 따라 크게 달라질 수 있습니다.

#### 2. 한 번 잘 나온 결과가 단순한 운일 수 있기 때문

예를 들어 seed 42에서만 hold-out이 상대적으로 쉬운 샘플들로 구성되어 좋은 점수가 나왔을 가능성을 배제할 수 없습니다.  
반대로 여러 seed에서 비슷하게 잘 나오면, 그 설정은 우연이 아니라 **상대적으로 신뢰할 수 있는 전략**이라고 볼 수 있습니다.

#### 3. 실험 결과의 안정성을 보기 위해서

좋은 방법은 한 번 최고 점수가 나온 방법이 아니라, **seed를 바꿔도 꾸준히 괜찮은 성능을 내는 방법**입니다.

#### 4. augmentation 효과를 제대로 비교하기 위해서

seed에 따라 결과가 크게 흔들리는 상태에서 augmentation까지 넣어버리면,

- 성능 변화가 augmentation 때문인지,
- 아니면 split 차이 때문인지

를 구분하기 어려워집니다.

#### 5. best checkpoint 선택의 신뢰성 확인

이번 실험에서는 dev hold-out 기준으로 best epoch를 고르는 구조였기 때문에, hold-out 구성이 seed마다 너무 달라지면 **best checkpoint 자체의 신뢰성**도 흔들릴 수 있었습니다.

---

### 실험 결과

#### 1) 각 seed의 best checkpoint epoch

- **seed 42**: epoch **11**
- **seed 7**: epoch **7**
- **seed 123**: epoch **9**

이 결과는 best 지점이 모두 초중반에 나온다는 뜻입니다.  
즉, 현재 설정은 후반 장기 학습으로 이득을 보는 구조라기보다, **비교적 이른 시점에서 hold-out 일반화 성능의 정점이 형성되는 구조**에 가까웠습니다.

#### 2) 각 seed의 최저 val_logloss

- **seed 42**: **0.217685**
- **seed 7**: **0.184079**
- **seed 123**: **0.215801**

hold-out 기준만 보면 **seed 7**이 가장 좋아 보였습니다.

#### 3) 각 seed의 최종 dev 일반화 결과 (`dev_full_logloss`)

- **seed 42**: **0.172517**
- **seed 7**: **0.189685**
- **seed 123**: **0.174990**

최종 dev 일반화 기준으로 보면:

- 가장 좋음: **seed 42**
- 그 다음: **seed 123**
- 가장 나쁨: **seed 7**

---

### 해석

이 실험에서 가장 중요했던 점은,

> **hold-out 최저값이 가장 낮다고 해서 전체 dev 일반화가 항상 가장 좋은 것은 아니다**

라는 사실을 확인한 것입니다.

즉, 특정 seed에서 hold-out 점수가 좋게 나왔다고 하더라도, 그것이 전체 동적 dev에 대한 일반화 성능으로 그대로 이어지지 않을 수 있었습니다.  
이는 작은 검증셋 환경에서 자주 발생하는 문제이며, 이번 실험에서는 그 차이가 꽤 분명하게 드러났습니다.

### 결론

이후 증강 실험에서는 **가장 최종 dev 일반화 성능이 좋았던 seed 42를 고정**해서 사용했습니다.  
이렇게 해야 이후의 synthetic augmentation 실험 결과를 보다 일관되게 비교할 수 있다고 판단했습니다.

## 5. 실험 3~5: synthetic 데이터 증강 실험

### 목적

이제 목표는 명확했습니다.

> **정적 train 데이터와 동적 dev/test 데이터 사이의 도메인 차이를 줄이기 위해, 평가 환경을 반영한 synthetic 데이터를 추가하자.**

핵심은 단순히 데이터를 많이 만드는 것이 아니라,

- dev/test에서 나타나는 시점 변화,
- 조명 변화,
- 그림자,
- 경계 조건 변화,
- 구조 스타일 변화

를 synthetic 이미지 안에 얼마나 잘 반영하느냐였습니다.

또한 이 단계부터는 **seed 42를 고정**하고, 동일한 best checkpoint 선택 코드 기준으로 비교했습니다.

## 6. Baseline: 증강 없는 기준 실험

### 설정

- 학습: **train 1000**
- validation: **dev100**
- 목적: 이후 synthetic 실험의 비교 기준점 확보

### 결과

- best dev_full_logloss: **0.079616**

### 해석

기본 2-view CNN baseline만으로도 어느 정도 의미 있는 성능은 확보할 수 있었습니다.  
하지만 train의 정적 환경과 dev/test의 동적 환경 사이 차이를 메우기에는 한계가 있었습니다.

### 장점

- 구조가 단순하고 재현이 쉬움
- 비교 기준점으로 적절함

### 한계

- 평가 환경 적응력이 부족함
- camera / light / domain gap을 직접 메우지 못함

## 7. synth100: 보수적 eval-style synthetic 시작

### 설정

- 학습: **train 1000 + synthetic 100**
- validation: **dev100**
- synthetic 방향:
  - 실제 train 이미지를 기반으로 한 **보수적 eval-style 증강**
  - camera perturbation
  - brightness / contrast / gamma
  - shadow / vignette
  - small boundary-like shift 일부 반영

### 결과

- best epoch: **35**
- best dev_full_logloss: **0.067929**

### baseline 대비 변화

- **0.079616 → 0.067929**
- baseline 대비 **명확한 개선**

### 왜 좋아졌는가?

완전히 새로운 구조를 자유롭게 생성하기보다, **실제 train 구조를 유지한 채** dev/test에서 중요한 변화 요소만 조심스럽게 반영했기 때문입니다.  
즉, 구조적 의미를 해치지 않으면서도 평가 환경의 시점·조명 변화를 학습에 노출시킨 것이 효과적이었습니다.

### 장점

- baseline보다 확실히 좋아짐
- 분포를 크게 망치지 않음
- synthetic가 실제로 도움이 된다는 첫 번째 근거를 제공

### 한계

- 다양성이 아직 부족함
- 구조 타입이 제한적임
- boundary 성격 보강이 충분히 정교하지 않음

---

## 8. synth300: 평가환경 적응형 synthetic 확대

### 설정

- 학습: **train 1000 + synthetic 300**
- validation: **dev100**
- synthetic 방향:
  - synth100 방향 유지
  - evalmix 강화
  - style 축 추가: `slender`, `compact`, `lean`, `step`
  - hard mode 일부 추가
  - 더 다양한 camera / light / shadow perturbation

### 결과

- best epoch: **27**
- best dev_full_logloss: **0.044445**

### synth100 대비 변화

- **0.067929 → 0.044445**
- 큰 폭 개선

### 왜 좋아졌는가?

이번 개선은 단순히 synthetic 개수를 늘려서 나온 결과라기보다,

- dev/test 적응에 필요한 변동성을 더 촘촘하게 넣고,
- 구조는 보수적으로 유지하면서도,
- 장면 다양성을 넓힌 설계

덕분이라고 해석할 수 있습니다.

즉, 무작정 자유 생성 범위를 넓힌 것이 아니라 **평가환경 적응형 변화 요소를 정교하게 확장한 것**이 효과적이었습니다.

### 장점

- 지금까지 실험 중 큰 폭의 개선
- “자유 생성 확대”보다 “보수적 synthetic 강화”가 더 효과적이라는 근거 제공
- camera / light perturbation 효과가 강하게 드러남

### 한계

- 생성 품질 필터가 아직 약함
- 일부 샘플은 서로 너무 유사하거나 단순할 가능성이 있음
- checkpoint resume / synthetic 파일 무결성 등 운영 이슈가 중간에 존재

---

## 9. synth500: 난이도 버킷과 스타일 균형 확장

### 설정

- 학습: **train 1000 + synthetic 500**
- validation: **dev100**
- synthetic 방향:
  - synth300 유지
  - 기존 300개 보존
  - 새 200개 추가
  - 난이도 버킷 추가:
    - `eval_easy`
    - `structure_perturb`
    - `boundary_lite`
    - `boundary_strong`
  - style 균형 추가:
    - `slender`
    - `compact`
    - `lean`
    - `step`
    - `wide`
  - 품질 필터
  - 중복 억제
  - source 품질 점수 반영

### 생성 분포

- 총 **500개**
- label 분포: **stable 250 / unstable 250**
- 추가 200개 분포:
  - `eval_easy`: 60
  - `structure_perturb`: 50
  - `boundary_lite`: 50
  - `boundary_strong`: 40

### 1차 결과

- best epoch: **56**
- best val_logloss: **0.03568568825721741**
- final dev_full_logloss: **0.035686**

### synth300 대비 변화

- **0.044445 → 0.035686**
- 추가 개선

### 왜 좋아졌는가?

synth300의 방향성을 유지하면서도,

- 난이도 설계를 더 명확하게 넣고,
- 구조 타입과 스타일 균형을 맞추고,
- 좋은 synthetic를 더 잘 추가하는 방식으로 확장했기 때문입니다.

즉, 단순 랜덤 추가가 아니라 **성능에 기여하는 synthetic 분포를 의도적으로 설계한 확장**이었습니다.

### 장점

- 당시 기준 최고 성능
- synth300의 장점을 유지하면서 추가 개선
- label 균형, 난이도 버킷, 스타일 균형이 명확함
- 제출용 결과까지 안정적으로 생성 완료

### 한계

- 후반 epoch에서 변동성이 큼
- 추가된 200개가 항상 안정적으로 이득만 주는 것은 아니었음
- 학습 후반부 일부 epoch에서 성능이 크게 흔들림
- synthetic 품질 관리를 더 정교하게 할 여지가 남아 있음

---

## 10. 후반 loss 급등 문제와 원인 분석

synth500 실험에서는 학습을 더 길게 진행했을 때 후반부에서 갑작스럽게 loss가 크게 튀는 현상이 관찰되었습니다.

예를 들어,

- **epoch 72에서 val_logloss가 0.8633까지 급등**

하는 현상이 발생했습니다.

처음에는 synthetic 데이터 개수 증가나 품질 문제로 보일 수 있었지만, 원인을 추적한 결과 핵심 문제는 다른 곳에 있었습니다.

### 왜 후반으로 갈수록 흔들렸는가?

핵심 원인은 **resume 이후 CosineAnnealingLR의 `T_max`가 이전 값(40)에 묶여 있던 점**이었습니다.

이 상태에서 학습을 더 길게 이어가면,

- scheduler가 의도한 형태로 매끄럽게 감소하지 않고,
- 후반부에 learning rate가 비정상적으로 다시 커지며,
- validation logloss가 급등하는 현상

이 발생했습니다.

즉, 이 문제는 단순히 “synthetic가 많아져서 불안정해졌다”기보다는,

> **resume + scheduler 상태 불일치 + 후반 과신(overconfidence)**

이 겹치면서 발생한 운영 이슈에 가까웠습니다.

## 11. 안정화 이후 최종 synth500 결과

후반 폭주 문제를 정리한 뒤, 최고 checkpoint는 유지하면서 학습 종료 전략을 안정화했습니다.

### 최종 안정화 결과

- best epoch: **66**
- best val_logloss: **0.02120145782828331**
- final dev_full_logloss: **0.021201**

또한,

- **early_stopping = 12**
- epoch **79**에서 조기 종료

를 적용해, 불필요한 후반 폭주 구간을 잘라냈습니다.

### 해석

이는 synth500의 방향 자체가 틀렸던 것이 아니라,

- 좋은 synthetic 설계는 유지하고,
- 학습 스케줄 관리와 종료 조건을 더 안정적으로 가져갔을 때
- 성능이 추가로 더 좋아질 수 있다는 점

을 보여준 결과였습니다.

## 12. 전체 성능 흐름 정리

최종적으로 주요 실험의 성능 흐름은 다음과 같습니다.

- **Baseline**: 0.079616
- **synth100**: 0.067929
- **synth300**: 0.044445
- **synth500 (초기 best)**: 0.035686
- **synth500 (안정화 최종 best)**: 0.021201

### 해석

이 흐름에서 가장 중요한 포인트는 다음과 같습니다.

1. **synthetic는 분명히 효과가 있었다.**  
   단, 단순 개수 증가가 아니라 **평가환경 적응형 synthetic**이어야 했다.

2. **자유 생성보다 보수적 강화가 더 잘 맞았다.**  
   이번 대회의 핵심은 train과 dev/test의 도메인 차이를 줄이는 것이었기 때문에, 구조를 무리하게 새로 발명하는 것보다 평가 환경 적응이 더 중요했다.

3. **synthetic 설계가 좋아질수록 성능이 일관되게 개선되었다.**  
   baseline → synth100 → synth300 → synth500으로 갈수록 성능이 지속적으로 좋아졌다는 점은, 이 방향성이 우연이 아니라 실제로 dev100 일반화에 유효했다는 근거가 된다.

4. **후반부 불안정성은 데이터 자체보다 학습 운영 이슈의 영향이 컸다.**  
   따라서 성능 향상을 위해서는 데이터 설계뿐 아니라 scheduler, resume 전략, early stopping 관리도 매우 중요했다.

## 13. 최종 결론

이번 대회 실험을 통해 얻은 가장 중요한 결론은 세 가지입니다.

### 1) synthetic 데이터는 실제로 효과가 있었다

다만 중요한 것은 단순한 양적 증가가 아니라, **동적 평가 환경에 맞춘 방향성 있는 synthetic 설계**였습니다.

### 2) 자유 생성보다 평가환경 적응형 보수적 강화가 더 적합했다

이번 대회에서는 구조 자체를 완전히 새롭게 만드는 것보다,

- 시점 변화
- 조명 변화
- 그림자
- 경계 조건 변화
- 구조 스타일 다양화

를 현실적으로 반영하는 방식이 더 효과적이었습니다.

### 3) 현재 최선 결과는 synth500 안정화 버전이다

실험 전 과정을 통틀어 현재 기준 최종적으로 가장 좋은 결과는 **synth500 안정화 버전 (`dev_full_logloss = 0.021201`)** 이었습니다.

## 15. 회고

이번 실험은 단순히 “데이터를 더 넣어서 성능을 올린 과정”이 아니었습니다.

- dev split 비율을 어떻게 잡을지,
- seed를 어떻게 고정할지,
- synthetic를 어떤 방향으로 설계할지,
- 학습 후반 불안정을 어떻게 제어할지

를 하나씩 검증해 가며 **신뢰할 수 있는 개선 경로를 쌓아간 과정**에 더 가까웠습니다.

특히 작은 dev 환경에서는,

- hold-out 설계,
- checkpoint 선택,
- seed 분산,
- scheduler 상태

같은 요소들이 모두 결과에 큰 영향을 줄 수 있음을 확인했습니다.

결과적으로 이번 대회에서는 **정적 train 데이터만으로는 부족했던 동적 평가 환경 적응을, synthetic 데이터 설계와 학습 운영 안정화로 보완하는 전략**이 가장 효과적이었습니다.
