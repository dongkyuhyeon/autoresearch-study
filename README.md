# 구조물 안전성 물리 추론 AI 경진대회 실험 기록

## final ranking 44/484

## 1. 대회 개요

이번 프로젝트는 **구조물 안전성 물리 추론 AI 경진대회**를 준비하면서 수행한 실험 과정과 시행착오, 그리고 최종적으로 얻은 성능 개선 결과를 정리한 문서입니다.  
대회의 핵심 과제는 이미지로부터 구조물의 안정성을 판단하는 것이었고, 학습 및 평가 데이터의 성격 차이가 매우 중요했습니다.

### 데이터 구성

- **train**: 정적(static) 이미지 데이터
- **dev / test**: 동적(dynamic) 이미지 데이터
- 모든 데이터는 **이미지 기반 데이터셋**으로 구성

즉, 학습에는 상대적으로 정적인 환경의 이미지가 주어졌고, 실제 성능을 평가하는 dev/test는 더 다양한 시점, 조명, 장면 변화가 포함된 **동적 도메인**이었습니다.  
따라서 이번 대회에서는 단순히 train 성능을 높이는 것보다, **정적 학습 데이터와 동적 평가 데이터 사이의 도메인 차이(domain gap)를 얼마나 잘 줄이느냐**가 핵심이었습니다.

---

## 2. 전체 실험 흐름 요약

실험은 크게 다음 흐름으로 진행되었습니다.

1. **dev 분할 비율 탐색**  
   동적 dev 데이터를 일부 학습에 사용하고, 나머지를 hold-out 검증용으로 두었을 때 어떤 비율이 가장 잘 일반화되는지 확인

2. **seed 고정 실험**  
   작은 dev 데이터셋 분할에서 발생하는 우연성과 분산을 줄이기 위해 seed별 결과를 비교하고, 이후 실험의 기준 seed를 결정

3. **synthetic 데이터 증강 실험**  
   train 이미지에 dev/test 조건을 반영한 synthetic 이미지를 점진적으로 추가하여 도메인 적응 성능 향상 여부를 검증

이 흐름은 단순히 성능 숫자만 비교한 것이 아니라,

- 어떤 split이 더 믿을 만한지,
- 어떤 seed가 더 안정적인지,
- synthetic 데이터가 실제로 dev/test 일반화에 도움이 되는지

를 순서대로 검증하는 과정이었습니다.

## 3. 실험 1: dev 분할 비율 비교

### 목적

처음 학습할 때는 동적 이미지를 더 정확하게 맞추기 위해, `joe_baseline_code_upgrade` 기반으로 dev 데이터를 학습용 일부와 hold-out 검증용 나머지로 나누어 사용했습니다.  
핵심 질문은 다음과 같았습니다.

> **dev를 어느 정도 비율로 학습에 포함시켜야 동적 도메인 적응과 검증 신뢰성을 동시에 확보할 수 있는가?**

비교한 설정은 다음 세 가지였습니다.

- 20/80
- 30/70
- 40/60

여기서 앞 숫자는 `train + dev 일부`를 이용한 학습 비중, 뒤 숫자는 `hold-out 검증용 dev` 비중입니다.

---

### 실험 결과

- **Experiment 1 (20/80)**  
  - best val_logloss = **0.2291**
  - 최종 dev_full_logloss = **0.185459**

- **Experiment 2 (30/70)**  
  - best val_logloss = **0.1701**
  - 최종 dev_full_logloss = **0.120270**

- **Experiment 3 (40/60)**  
  - best val_logloss = **0.2304**
  - 최종 dev_full_logloss = **0.160855**

### 결론

세 실험 중 **Experiment 2 (30/70)** 이 가장 좋은 결과를 보였습니다.

---

### 해석

#### 왜 30/70이 가장 좋았는가?

`train 1000 + dev 30` 설정은 동적 도메인 적응에 필요한 최소한 이상의 dev 정보를 학습에 제공하면서도, 동시에 `dev 70`이라는 비교적 충분한 hold-out 검증 세트를 유지할 수 있었습니다.

즉,

- 학습 측면에서는 동적 데이터 특성을 어느 정도 반영할 수 있었고,
- 검증 측면에서는 checkpoint 선택에 필요한 일반화 판단 근거를 충분히 확보할 수 있었습니다.

#### 왜 20/80은 상대적으로 불리했는가?

- dev 20은 학습에 넣기에는 너무 적었습니다.
- 동적 도메인 적응을 하기에는 정보량이 부족했습니다.
- hold-out은 넉넉했지만, 정작 모델이 동적 데이터 특성을 배우는 데 한계가 있었습니다.

#### 왜 40/60은 기대보다 덜 좋았는가?

- dev 40은 학습에는 더 도움이 될 수 있습니다.
- 하지만 hold-out 크기가 줄어들면서 checkpoint 선택의 신뢰성이 떨어질 수 있습니다.
- 작은 dev 환경에서는 검증셋이 지나치게 줄어들면, 오히려 generalization 판단이 불안정해질 수 있습니다.

### 정리

이 실험을 통해, 단순히 dev를 많이 학습에 넣는다고 항상 좋은 것이 아니라 **학습 적응 효과와 검증 신뢰성 사이의 균형**이 중요하다는 점을 확인했습니다.  
이후 실험들은 **30/70 분할 전략**을 기본 방향으로 삼았습니다.

## 4. 실험 2: seed 고정 및 분할 안정성 검증

### 목적

다음 단계 목표는 **동적 데이터를 더 잘 맞추기 위한 데이터 증강**이었습니다.  
하지만 증강 실험을 진행하기 전에 먼저 확인해야 할 것이 있었습니다.

> **현재 성능 차이가 정말 augmentation 때문인지, 아니면 seed에 따라 달라진 분할 운 때문인지 구분할 수 있는가?**

그래서 증강 실험에 들어가기 전, seed를 고정하기 위한 검증을 먼저 수행했습니다.

---

### 왜 seed를 고정해야 했는가?

#### 1. 데이터 분할 결과가 seed에 따라 달라지기 때문

특히 dev 100처럼 작은 데이터셋을 30/70으로 나누면,

- 어떤 샘플이 train 쪽으로 들어가고
- 어떤 샘플이 hold-out으로 들어가는지

가 seed에 따라 크게 달라질 수 있습니다.

#### 2. 한 번 잘 나온 결과가 단순한 운일 수 있기 때문

예를 들어 seed 42에서만 hold-out이 상대적으로 쉬운 샘플들로 구성되어 좋은 점수가 나왔을 가능성을 배제할 수 없습니다.  
반대로 여러 seed에서 비슷하게 잘 나오면, 그 설정은 우연이 아니라 **상대적으로 신뢰할 수 있는 전략**이라고 볼 수 있습니다.

#### 3. 실험 결과의 안정성을 보기 위해서

좋은 방법은 한 번 최고 점수가 나온 방법이 아니라, **seed를 바꿔도 꾸준히 괜찮은 성능을 내는 방법**입니다.

#### 4. augmentation 효과를 제대로 비교하기 위해서

seed에 따라 결과가 크게 흔들리는 상태에서 augmentation까지 넣어버리면,

- 성능 변화가 augmentation 때문인지,
- 아니면 split 차이 때문인지

를 구분하기 어려워집니다.

#### 5. best checkpoint 선택의 신뢰성 확인

이번 실험에서는 dev hold-out 기준으로 best epoch를 고르는 구조였기 때문에, hold-out 구성이 seed마다 너무 달라지면 **best checkpoint 자체의 신뢰성**도 흔들릴 수 있었습니다.

---

### 실험 결과

#### 1) 각 seed의 best checkpoint epoch

- **seed 42**: epoch **11**
- **seed 7**: epoch **7**
- **seed 123**: epoch **9**

이 결과는 best 지점이 모두 초중반에 나온다는 뜻입니다.  
즉, 현재 설정은 후반 장기 학습으로 이득을 보는 구조라기보다, **비교적 이른 시점에서 hold-out 일반화 성능의 정점이 형성되는 구조**에 가까웠습니다.

#### 2) 각 seed의 최저 val_logloss

- **seed 42**: **0.217685**
- **seed 7**: **0.184079**
- **seed 123**: **0.215801**

hold-out 기준만 보면 **seed 7**이 가장 좋아 보였습니다.

#### 3) 각 seed의 최종 dev 일반화 결과 (`dev_full_logloss`)

- **seed 42**: **0.172517**
- **seed 7**: **0.189685**
- **seed 123**: **0.174990**

최종 dev 일반화 기준으로 보면:

- 가장 좋음: **seed 42**
- 그 다음: **seed 123**
- 가장 나쁨: **seed 7**

---

### 해석

이 실험에서 가장 중요했던 점은,

> **hold-out 최저값이 가장 낮다고 해서 전체 dev 일반화가 항상 가장 좋은 것은 아니다**

라는 사실을 확인한 것입니다.

즉, 특정 seed에서 hold-out 점수가 좋게 나왔다고 하더라도, 그것이 전체 동적 dev에 대한 일반화 성능으로 그대로 이어지지 않을 수 있었습니다.  
이는 작은 검증셋 환경에서 자주 발생하는 문제이며, 이번 실험에서는 그 차이가 꽤 분명하게 드러났습니다.

### 결론

이후 증강 실험에서는 **가장 최종 dev 일반화 성능이 좋았던 seed 42를 고정**해서 사용했습니다.  
이렇게 해야 이후의 synthetic augmentation 실험 결과를 보다 일관되게 비교할 수 있다고 판단했습니다.

## 5. 실험 3~5: synthetic 데이터 증강 실험

### 목적

이제 목표는 명확했습니다.

> **정적 train 데이터와 동적 dev/test 데이터 사이의 도메인 차이를 줄이기 위해, 평가 환경을 반영한 synthetic 데이터를 추가하자.**

핵심은 단순히 데이터를 많이 만드는 것이 아니라,

- dev/test에서 나타나는 시점 변화,
- 조명 변화,
- 그림자,
- 경계 조건 변화,
- 구조 스타일 변화

를 synthetic 이미지 안에 얼마나 잘 반영하느냐였습니다.

또한 이 단계부터는 **seed 42를 고정**하고, 동일한 best checkpoint 선택 코드 기준으로 비교했습니다.

## 6. Baseline: 증강 없는 기준 실험

### 설정

- 학습: **train 1000**
- validation: **dev100**
- 목적: 이후 synthetic 실험의 비교 기준점 확보

### 결과

- best dev_full_logloss: **0.079616**

### 해석

기본 2-view CNN baseline만으로도 어느 정도 의미 있는 성능은 확보할 수 있었습니다.  
하지만 train의 정적 환경과 dev/test의 동적 환경 사이 차이를 메우기에는 한계가 있었습니다.

### 장점

- 구조가 단순하고 재현이 쉬움
- 비교 기준점으로 적절함

### 한계

- 평가 환경 적응력이 부족함
- camera / light / domain gap을 직접 메우지 못함

## 7. synth100: 보수적 eval-style synthetic 시작

### 설정

- 학습: **train 1000 + synthetic 100**
- validation: **dev100**
- synthetic 방향:
  - 실제 train 이미지를 기반으로 한 **보수적 eval-style 증강**
  - camera perturbation
  - brightness / contrast / gamma
  - shadow / vignette
  - small boundary-like shift 일부 반영

### 결과

- best epoch: **35**
- best dev_full_logloss: **0.067929**

### baseline 대비 변화

- **0.079616 → 0.067929**
- baseline 대비 **명확한 개선**

### 왜 좋아졌는가?

완전히 새로운 구조를 자유롭게 생성하기보다, **실제 train 구조를 유지한 채** dev/test에서 중요한 변화 요소만 조심스럽게 반영했기 때문입니다.  
즉, 구조적 의미를 해치지 않으면서도 평가 환경의 시점·조명 변화를 학습에 노출시킨 것이 효과적이었습니다.

### 장점

- baseline보다 확실히 좋아짐
- 분포를 크게 망치지 않음
- synthetic가 실제로 도움이 된다는 첫 번째 근거를 제공

### 한계

- 다양성이 아직 부족함
- 구조 타입이 제한적임
- boundary 성격 보강이 충분히 정교하지 않음

---

## 8. synth300: 평가환경 적응형 synthetic 확대

### 설정

- 학습: **train 1000 + synthetic 300**
- validation: **dev100**
- synthetic 방향:
  - synth100 방향 유지
  - evalmix 강화
  - style 축 추가: `slender`, `compact`, `lean`, `step`
  - hard mode 일부 추가
  - 더 다양한 camera / light / shadow perturbation

### 결과

- best epoch: **27**
- best dev_full_logloss: **0.044445**

### synth100 대비 변화

- **0.067929 → 0.044445**
- 큰 폭 개선

### 왜 좋아졌는가?

이번 개선은 단순히 synthetic 개수를 늘려서 나온 결과라기보다,

- dev/test 적응에 필요한 변동성을 더 촘촘하게 넣고,
- 구조는 보수적으로 유지하면서도,
- 장면 다양성을 넓힌 설계

덕분이라고 해석할 수 있습니다.

즉, 무작정 자유 생성 범위를 넓힌 것이 아니라 **평가환경 적응형 변화 요소를 정교하게 확장한 것**이 효과적이었습니다.

### 장점

- 지금까지 실험 중 큰 폭의 개선
- “자유 생성 확대”보다 “보수적 synthetic 강화”가 더 효과적이라는 근거 제공
- camera / light perturbation 효과가 강하게 드러남

### 한계

- 생성 품질 필터가 아직 약함
- 일부 샘플은 서로 너무 유사하거나 단순할 가능성이 있음
- checkpoint resume / synthetic 파일 무결성 등 운영 이슈가 중간에 존재

---

## 9. synth500: 난이도 버킷과 스타일 균형 확장

### 설정

- 학습: **train 1000 + synthetic 500**
- validation: **dev100**
- synthetic 방향:
  - synth300 유지
  - 기존 300개 보존
  - 새 200개 추가
  - 난이도 버킷 추가:
    - `eval_easy`
    - `structure_perturb`
    - `boundary_lite`
    - `boundary_strong`
  - style 균형 추가:
    - `slender`
    - `compact`
    - `lean`
    - `step`
    - `wide`
  - 품질 필터
  - 중복 억제
  - source 품질 점수 반영

### 생성 분포

- 총 **500개**
- label 분포: **stable 250 / unstable 250**
- 추가 200개 분포:
  - `eval_easy`: 60
  - `structure_perturb`: 50
  - `boundary_lite`: 50
  - `boundary_strong`: 40

### 1차 결과

- best epoch: **56**
- best val_logloss: **0.03568568825721741**
- final dev_full_logloss: **0.035686**

### synth300 대비 변화

- **0.044445 → 0.035686**
- 추가 개선

### 왜 좋아졌는가?

synth300의 방향성을 유지하면서도,

- 난이도 설계를 더 명확하게 넣고,
- 구조 타입과 스타일 균형을 맞추고,
- 좋은 synthetic를 더 잘 추가하는 방식으로 확장했기 때문입니다.

즉, 단순 랜덤 추가가 아니라 **성능에 기여하는 synthetic 분포를 의도적으로 설계한 확장**이었습니다.

### 장점

- 당시 기준 최고 성능
- synth300의 장점을 유지하면서 추가 개선
- label 균형, 난이도 버킷, 스타일 균형이 명확함
- 제출용 결과까지 안정적으로 생성 완료

### 한계

- 후반 epoch에서 변동성이 큼
- 추가된 200개가 항상 안정적으로 이득만 주는 것은 아니었음
- 학습 후반부 일부 epoch에서 성능이 크게 흔들림
- synthetic 품질 관리를 더 정교하게 할 여지가 남아 있음

---

## 10. 후반 loss 급등 문제와 원인 분석

synth500 실험에서는 학습을 더 길게 진행했을 때 후반부에서 갑작스럽게 loss가 크게 튀는 현상이 관찰되었습니다.

예를 들어,

- **epoch 72에서 val_logloss가 0.8633까지 급등**

하는 현상이 발생했습니다.

처음에는 synthetic 데이터 개수 증가나 품질 문제로 보일 수 있었지만, 원인을 추적한 결과 핵심 문제는 다른 곳에 있었습니다.

### 왜 후반으로 갈수록 흔들렸는가?

핵심 원인은 **resume 이후 CosineAnnealingLR의 `T_max`가 이전 값(40)에 묶여 있던 점**이었습니다.

이 상태에서 학습을 더 길게 이어가면,

- scheduler가 의도한 형태로 매끄럽게 감소하지 않고,
- 후반부에 learning rate가 비정상적으로 다시 커지며,
- validation logloss가 급등하는 현상

이 발생했습니다.

즉, 이 문제는 단순히 “synthetic가 많아져서 불안정해졌다”기보다는,

> **resume + scheduler 상태 불일치 + 후반 과신(overconfidence)**

이 겹치면서 발생한 운영 이슈에 가까웠습니다.

## 11. 안정화 이후 최종 synth500 결과

후반 폭주 문제를 정리한 뒤, 최고 checkpoint는 유지하면서 학습 종료 전략을 안정화했습니다.

### 최종 안정화 결과

- best epoch: **66**
- best val_logloss: **0.02120145782828331**
- final dev_full_logloss: **0.021201**

또한,

- **early_stopping = 12**
- epoch **79**에서 조기 종료

를 적용해, 불필요한 후반 폭주 구간을 잘라냈습니다.

### 해석

이는 synth500의 방향 자체가 틀렸던 것이 아니라,

- 좋은 synthetic 설계는 유지하고,
- 학습 스케줄 관리와 종료 조건을 더 안정적으로 가져갔을 때
- 성능이 추가로 더 좋아질 수 있다는 점

을 보여준 결과였습니다.

## 12. 전체 성능 흐름 정리

최종적으로 주요 실험의 성능 흐름은 다음과 같습니다.

- **Baseline**: 0.079616
- **synth100**: 0.067929
- **synth300**: 0.044445
- **synth500 (초기 best)**: 0.035686
- **synth500 (안정화 최종 best)**: 0.021201

### 해석

이 흐름에서 가장 중요한 포인트는 다음과 같습니다.

1. **synthetic는 분명히 효과가 있었다.**  
   단, 단순 개수 증가가 아니라 **평가환경 적응형 synthetic**이어야 했다.

2. **자유 생성보다 보수적 강화가 더 잘 맞았다.**  
   이번 대회의 핵심은 train과 dev/test의 도메인 차이를 줄이는 것이었기 때문에, 구조를 무리하게 새로 발명하는 것보다 평가 환경 적응이 더 중요했다.

3. **synthetic 설계가 좋아질수록 성능이 일관되게 개선되었다.**  
   baseline → synth100 → synth300 → synth500으로 갈수록 성능이 지속적으로 좋아졌다는 점은, 이 방향성이 우연이 아니라 실제로 dev100 일반화에 유효했다는 근거가 된다.

4. **후반부 불안정성은 데이터 자체보다 학습 운영 이슈의 영향이 컸다.**  
   따라서 성능 향상을 위해서는 데이터 설계뿐 아니라 scheduler, resume 전략, early stopping 관리도 매우 중요했다.

## Experiment 1. Blend99

## 개요
`blend99`는 별도의 추가 학습으로 얻은 새로운 모델이 아니라, 이미 학습이 완료된 두 모델의 예측 확률을 **추론 단계에서 가중 평균한 제출본**이다.

사용된 두 모델은 다음과 같다.

- `synth500 best checkpoint`
- `synth300 best checkpoint`

최종 확률은 아래 식으로 계산했다.

```text
final_prob = 0.99 * prob_synth500 + 0.01 * prob_synth300
```

즉, `synth500`의 예측을 거의 그대로 유지하면서, `synth300`의 예측을 1%만 반영해 확률을 아주 미세하게 보정하는 방식이다.

---

## blend99는 재학습 모델이 아님
이 실험에서 중요한 점은 `blend99` 자체가 **새롭게 학습된 모델이 아니라는 것**이다.

과정은 다음과 같았다.

1. `synth500 best checkpoint`로 dev/test에 대한 예측 확률 생성
2. `synth300 best checkpoint`로 동일한 샘플에 대한 예측 확률 생성
3. 두 확률을 `99:1` 비율로 가중 평균
4. 그 결과를 최종 `submission.csv`에 사용

이 과정에서는 optimizer 업데이트, backward, epoch 반복 같은 학습 과정이 전혀 없었다. 따라서 `blend99`는 **모델 블렌딩 결과물**이지, 새로운 학습 산출물이 아니다.

---

## 왜 99:1 비율을 사용했는가
핵심 이유는 `synth500`이 이미 가장 강한 단일 모델이었기 때문이다. 따라서 블렌딩의 목적은 `synth300`의 성능을 크게 반영하는 것이 아니라, `synth500`의 **과도한 확신(overconfidence)** 을 아주 소폭 완화하는 데 있었다.

`raw500`은 dev LogLoss 기준으로는 가장 낮은 값을 기록했지만, 확률 분포가 지나치게 극단적인 문제가 있었다. 특히 매우 작은 확률값(`<= 1e-3`)에 과하게 몰리는 비율이 높아, 실제 테스트 환경에서는 잘못된 극단 예측이 큰 손실로 이어질 위험이 있었다.

이때 99:1 블렌드는 다음과 같은 장점을 가졌다.

- `synth500`의 높은 성능을 거의 그대로 유지할 수 있음
- 극단적인 저확률/고확률 예측을 아주 조금 완화할 수 있음
- dev 성능 손실을 최소화하면서 제출 안정성을 높일 수 있음

반면 95:5처럼 더 큰 비율 보정은 확률을 더 부드럽게 만들 수 있었지만, 그만큼 강한 단일 모델인 `synth500`의 예측을 더 흐릴 가능성이 있었다. 따라서 `99:1`은 **가장 작은 수준의 실전형 절충안**으로 판단했다.

---

## 결과 해석
숫자 기준으로 보면 dev LogLoss 최고는 여전히 `raw500`이었다.

- `raw500`: 0.0212699
- `blend99`: 0.0213717

즉 `blend99`가 dev 점수만으로 `raw500`을 이긴 것은 아니다. 그럼에도 `blend99`를 최종본 후보로 본 이유는 다음과 같다.

- `raw500`은 숫자는 최고지만 과신 리스크가 컸음
- `blend99`는 dev 손실을 거의 희생하지 않으면서 과신을 줄였음
- 따라서 리더보드 환경에서는 더 안정적인 제출 전략이 될 가능성이 있었음

이를 한 문장으로 정리하면 다음과 같다.

> `raw500`은 dev 점수 최고 모델이었고, `blend99`는 실전 제출 안정성까지 고려한 절충형 최종본이었다.

---

## 정리
`blend99`는 별도의 추가 학습 모델이 아니라, `synth500 best`와 `synth300 best`의 예측을 추론 단계에서 `99:1`로 섞은 전역 블렌드 방식이다. 이 전략은 `synth500`의 강한 성능은 유지하면서도, 과도한 확신 분포를 아주 소폭 완화해 실제 제출 리스크를 줄이기 위한 목적에서 사용되었다.

---

## Experiment 2. Synth1000 성능 저하 해석

## 개요
synthetic 데이터를 더 많이 넣으면 과적합이 줄고 성능이 계속 좋아질 것이라고 예상했지만, 실제 실험에서는 `train 1000 + synthetic 1000` 설정이 `synthetic 500`보다 오히려 성능이 나빠졌다.

이 결과는 단순히 데이터 수가 많아지면 무조건 일반화가 좋아지는 것이 아니라, **추가된 데이터의 품질과 분포 적합성**이 훨씬 더 중요하다는 점을 보여준다.

---

## 왜 synthetic를 많이 넣었는데도 성능이 떨어질 수 있는가

### 1. synthetic 비중이 너무 커지면서 실제 train 분포를 덮어버릴 수 있음
`train 1000 + synthetic 1000`은 실제 데이터와 synthetic 데이터가 거의 1:1 비율이다. 이 경우 synthetic 품질이 충분히 높고 실제 dev/test 분포와 잘 맞으면 도움이 되지만, 그렇지 않으면 모델이 synthetic 쪽 특유의 패턴을 과하게 학습할 수 있다.

예를 들어 다음과 같은 synthetic 고유 특성이 누적될 수 있다.

- 렌더링 질감
- 그림자 패턴
- 카메라 왜곡 방식
- 구조 배치 습관
- stable/unstable 생성 규칙의 인위성

이런 요소가 많아지면 모델은 구조 안정성의 본질보다 **synthetic가 가진 생성 흔적**을 더 학습하게 될 위험이 있다.

---

### 2. 많은 증강과 좋은 증강은 다름
`synth500`에서 좋은 결과가 나온 이유는 단순히 개수가 많아서가 아니라, 실제 평가 환경을 겨냥해 **필요한 방향의 변동성**을 넣었기 때문이다.

예를 들면 다음 요소들이 효과적이었다.

- eval 환경 적응형 설계
- camera/light perturbation
- 보수적 구조 유지
- style 축과 난이도 설계

반면 `synth1000`이 성능이 떨어졌다면, 이는 보통 다음 상황으로 해석할 수 있다.

- 일반화에 도움 되는 샘플보다 불필요한 샘플이 너무 많이 들어감
- 좋은 synthetic 비율이 상대적으로 낮아짐
- 중복 또는 유사 샘플이 많아짐
- 품질이 낮은 synthetic까지 대량 포함됨

즉 `500개`까지는 유효한 다양성 증가였지만, `1000개`부터는 **쓸데없는 다양성 또는 잘못된 다양성**까지 커졌을 가능성이 높다.

---

### 3. 작은 데이터셋 환경에서는 synthetic 과다 주입이 더 위험함
이번 대회는 원래 데이터 규모가 매우 큰 편이 아니었다. 이런 환경에서는 synthetic의 미세한 공통 특성도 모델이 쉽게 배워버린다.

그 결과 모델은 실제 test에서 필요한 판단 기준보다, synthetic 데이터가 반복적으로 보여준 패턴에 더 민감해질 수 있다. 이는 overfitting과는 조금 다른 형태의 문제로, **잘못된 분포에 적응하는 현상**이라고 볼 수 있다.

---

### 4. 클래스 균형보다 난이도 균형이 더 중요할 수 있음
겉으로 보기에는 stable/unstable 비율이 맞더라도, 실제로는 난이도 분포가 치우쳐 있을 수 있다.

예를 들어 다음과 같은 문제가 있을 수 있다.

- 너무 쉬운 unstable 샘플이 많음
- boundary case가 부족함
- 특정 구조 스타일이 반복됨
- 한쪽 스타일에만 편향된 증강이 많음

이 경우 모델은 자신감은 높아질 수 있지만, 실제 경계 상황에 대한 일반화 능력은 오히려 떨어질 수 있다.

---

## 이번 실험의 해석
실험 흐름을 보면 다음과 같이 정리할 수 있다.

- `synth100`: 적응 신호가 부족함
- `synth300`: 큰 폭으로 개선
- `synth500`: 최적점에 가까운 성능
- `synth1000`: 과도한 synthetic 주입으로 역효과 발생 가능

즉 이번 문제에서는 **synthetic를 많이 넣는 것 자체가 목표가 아니라**, 실제 평가 환경에 맞는 synthetic를 **적절한 비율로 넣는 것**이 핵심이었다. 그 균형점이 `synthetic 500` 부근에 있었던 것으로 해석할 수 있다.

---

## 결론
`train 1000 + synthetic 1000`이 `synthetic 500`보다 성능이 낮았던 이유는, 추가된 synthetic가 일반화에 필요한 유효 변동성을 늘리기보다 synthetic 고유 편향, 중복, 저품질 패턴까지 함께 확대했기 때문으로 볼 수 있다.

이를 한 문장으로 정리하면 다음과 같다.

> 이번 실험에서는 “synthetic는 많을수록 좋다”가 아니라, “실제 평가환경에 맞는 synthetic를 적절한 품질과 비율로 넣는 것이 중요하다”는 점이 확인되었다.

---

## Experiment 3. 선택적 블렌드

## 개요
`blend99`보다 더 높은 성능이 나온 이유는, 모든 샘플을 동일하게 조금씩 섞는 방식에서 벗어나 **정말 위험한 샘플만 골라 강하게 보정하는 방식**으로 블렌딩 전략을 바꿨기 때문이다.

기존 `blend99`는 전역 블렌드였다. 즉 전체 샘플에 대해 아래 식을 일괄 적용했다.

```text
final_prob = 0.99 * prob_synth500 + 0.01 * prob_synth300
```

반면 새 최종본은 대부분의 샘플에서는 `synth500` 예측을 그대로 유지하고, `synth500`의 확률이 `0.2 ~ 0.8` 사이인 **애매한 샘플**에 대해서만 선택적으로 강한 보정을 적용했다.

해당 구간에서는 다음과 같이 더 강한 비율을 사용했다.

```text
final_prob = 0.3 * prob_synth500 + 0.7 * prob_synth300
```

---

## 기존 blend99 방식의 특징
`blend99`의 장점은 분명했다.

- 전체 샘플에 대해 동일하게 적용할 수 있어 구현이 단순함
- `synth500`의 성능을 거의 그대로 유지할 수 있음
- 과신을 아주 약하게 줄일 수 있음

하지만 한계도 있었다.

- 확실한 샘플과 애매한 샘플을 구분하지 못함
- 잘 맞히는 쉬운 샘플도 동일하게 보정됨
- 실제로 문제가 되는 경계 샘플에는 보정 강도가 부족할 수 있음

즉 `blend99`는 안정적인 전역 보정 방식이었지만, **보정이 필요한 샘플과 필요하지 않은 샘플을 구분하지 못한다는 비효율**이 있었다.

---

## 새 최종본의 핵심 아이디어
새 최종본의 핵심은 **선택적 블렌드(selective blend)** 였다.

기본 가정은 다음과 같았다.

- `synth500`은 확실한 샘플에서는 매우 강하다.
- 문제는 일부 경계 샘플에서의 과신 또는 오판 가능성이다.
- 따라서 모든 샘플을 일괄적으로 건드릴 필요는 없다.
- 오히려 애매한 샘플만 골라 더 강하게 보정하는 것이 효율적이다.

이 전략에서는 `synth500` 확률이 `0.2 ~ 0.8`인 샘플을 **불확실 구간**으로 정의했다. 이 구간은 모델이 확신이 부족하거나 경계 사례일 가능성이 높다고 판단한 영역이다. 따라서 이 샘플들에 대해서만 `synth300`의 비중을 크게 반영했다.

---

## 왜 성능이 더 좋아졌는가
이 방식이 더 좋았던 이유는 다음과 같다.

### 1. 확실한 샘플은 그대로 두었기 때문
`blend99`는 쉬운 샘플도, 어려운 샘플도 모두 조금씩 섞었다. 반면 새 방식은 확실한 샘플에서는 `synth500`을 그대로 사용했기 때문에, 원래 강한 단일 모델의 장점을 최대한 보존할 수 있었다.

### 2. 애매한 샘플에서만 강하게 보정했기 때문
실제로 문제가 되는 것은 일부 경계 샘플이었다. 이 샘플들에서는 `synth500`의 과신 또는 오판이 손실에 더 크게 영향을 줄 수 있었다. 따라서 이런 구간에 한해서 `synth300`을 더 강하게 섞는 것이 효과적이었다.

### 3. 불필요한 성능 손실을 줄였기 때문
전역 블렌드는 전체 샘플에 작은 변형을 주기 때문에, 원래 잘 맞히던 샘플의 예측도 일부 흐려질 수 있다. 선택적 블렌드는 필요한 샘플만 수정하므로 불필요한 손실을 줄이고, 실제로 중요한 샘플에서만 개선 효과를 낼 수 있었다.

---

## 결과
수치상으로도 차이가 분명했다.

- `blend99`: 0.0213717
- 새 최종본: 0.0118234

즉 선택적 블렌드 전략이 전역 블렌드보다 훨씬 더 낮은 dev LogLoss를 기록했다.

---

## 결론
이번 실험에서 확인된 핵심은 다음과 같다.

- 전역적으로 약하게 섞는 방식보다
- 애매한 구간만 선택적으로 강하게 보정하는 방식이 더 효과적이었다.

이를 한 문장으로 정리하면 다음과 같다.

> `blend99`는 전체 샘플을 조금씩 보정한 전역 블렌드였고, 새 최종본은 경계 샘플만 골라 강하게 보정한 선택적 블렌드였다. 그 결과 확실한 샘플에서는 `synth500`의 강점을 유지하고, 애매한 샘플에서는 오판 리스크를 줄여 더 낮은 dev LogLoss를 달성할 수 있었다.

## Experiment 4. Blend99의 dev loss 의미

## 개요
`blend99`의 dev loss가 계산되었다고 해서, `blend99`가 새롭게 학습된 모델이라는 뜻은 아니다. 여기서의 dev loss는 **학습 loss**가 아니라, 블렌딩된 예측 결과를 dev 정답과 비교해서 계산한 **평가용 LogLoss**이다.

---

## 학습 loss와 평가 loss의 차이

### 1. 학습 loss
학습 loss는 모델의 가중치를 업데이트하는 과정에서 사용된다.

흐름은 다음과 같다.

1. 입력 데이터를 모델에 넣음
2. 예측값 생성
3. 정답과 비교하여 loss 계산
4. backward 수행
5. optimizer로 가중치 업데이트

즉 학습 loss는 모델 파라미터를 실제로 바꾸는 데 사용되는 값이다.

### 2. 평가 loss
평가 loss는 이미 학습이 끝난 모델의 예측값을 정답과 비교해, 성능을 수치로 측정하는 값이다.

이 과정에서는 다음만 일어난다.

1. 모델이 예측 수행
2. 예측값과 정답 비교
3. LogLoss 계산

즉 가중치 업데이트는 전혀 일어나지 않는다.

---

## blend99의 dev loss는 어떻게 계산되었는가
`blend99`에서는 먼저 두 모델이 dev 데이터에 대해 각각 예측을 만든다.

- `prob_synth500`
- `prob_synth300`

그리고 아래 식으로 샘플별 최종 확률을 만든다.

```text
final_prob = 0.99 * prob_synth500 + 0.01 * prob_synth300
```

그 다음, 이 `final_prob`를 dev의 정답 라벨과 비교해서 LogLoss를 계산한다. 이 값이 바로 `blend99`의 dev loss이다.

즉 `0.0213717`이라는 숫자는 **새 모델이 학습되어 나온 loss가 아니라**, 두 모델의 예측을 섞은 결과를 dev에서 채점한 성능 지표이다.

---

## 왜 헷갈리기 쉬운가
보통 `loss`라는 단어를 학습 과정에서 많이 보기 때문에, 숫자가 나오면 자연스럽게 “학습했다”라고 느끼기 쉽다. 하지만 실제로는 다음처럼 구분해야 한다.

- `train loss`: 학습 중 가중치 업데이트에 사용되는 loss
- `val loss`: epoch마다 검증셋에 대해 계산하는 평가 loss
- `dev_full_logloss`: 최종 모델 또는 후처리 결과를 dev 전체에 대해 채점한 성능 지표
- `blend99 dev loss`: 블렌딩된 예측 결과를 dev에서 평가한 점수

따라서 `blend99`에서 dev loss가 존재한다는 사실은, 그 결과를 **평가했다는 뜻**이지 **재학습했다는 뜻은 아니다.**

---

## 결론
`blend99`의 dev loss는 모델을 새롭게 학습해서 나온 손실값이 아니라, 이미 학습된 `synth500`과 `synth300`의 예측을 블렌딩한 뒤 dev 정답과 비교해 계산한 평가 점수이다.
